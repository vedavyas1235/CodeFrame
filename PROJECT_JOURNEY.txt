PROJECT JOURNEY — HTML ANIMATION → MP4 EXPORTER
================================================================================

Last updated: 2026-05-27

--------------------------------------------------------------------------------
1. THE CORE IDEA (what I'm trying to build)
--------------------------------------------------------------------------------

I want a web app where:

  1. The user uploads (or pastes) a self-contained HTML file that contains
     some kind of animation — could be CSS animation, SVG, DOM/JS animation,
     a <canvas> 2D animation, or a WebGL/Three.js scene.
  2. The app renders that HTML inside a preview iframe so the user can see
     it running live.
  3. The user picks an export resolution (e.g. 720p / 1080p), frame rate
     (e.g. 30 / 60 fps) and duration (e.g. 5–30 seconds).
  4. The user clicks "Export" and the app produces a real MP4 video file
     of that animation — exactly matching what they see in the preview —
     entirely in the browser (no server, no ffmpeg.wasm round-trip if
     avoidable, no external service).
  5. They download the MP4.

Two capture "modes" are exposed:

  - QUICK MODE      → records the animation in real time (what you see is
                      what you get). Should work for ANY animation type,
                      including canvas/WebGL.
  - HIGH QUALITY    → frame-stepped / deterministic. Injects a virtual
    (FRAME-STEPPED)   clock into the iframe so each frame is rendered at
                      an exact timestamp. Smoother, no dropped frames —
                      but only works for animations that respect the
                      virtualized time APIs (rAF, setTimeout, Date.now,
                      performance.now).

Tech stack actually in use:
  - TanStack Start (React 19 + Vite 7) for the app shell
  - html2canvas for DOM rasterization
  - WebCodecs (VideoEncoder) + mp4-muxer for in-browser MP4 encoding
  - A custom virtual-time shim injected into the iframe for High Quality mode
  - A WebGL preserveDrawingBuffer shim injected for Quick mode

--------------------------------------------------------------------------------
2. WHY THIS IS HARD (the real problems under the hood)
--------------------------------------------------------------------------------

  - html2canvas rasterizes the DOM by re-parsing CSS — it CANNOT read the
    pixels of a <canvas> or <video> element directly. So for WebGL / 2D
    canvas animations, naive html2canvas snapshots come out blank or static.
  - WebGL contexts default to preserveDrawingBuffer:false, which means
    after the GPU composites a frame, the back-buffer is cleared.
    drawImage(canvas, …) then reads black/empty pixels.
  - html2canvas is SLOW (often 100–400ms per frame at 1080p). While it
    runs, it blocks the main thread, which means the iframe's rAF loop
    can't tick. Naive real-time capture therefore produces near-identical
    frames.
  - Virtual-time shims can't catch every animation API — Web Audio
    scheduling, libraries that cached the original setTimeout before the
    shim ran, GPU-driven shader time uniforms, etc.
  - WebCodecs encoders can emit chunks out of presentation order; manual
    timestamp handling is fragile.
  - At 1080p60 for long durations, memory + CPU pressure causes the tab
    to reload or freeze.

4. CURRENT KNOWN-GOOD MATRIX (what reliably exports today)
--------------------------------------------------------------------------------

                          | Quick mode | High Quality (frame-stepped)
--------------------------+------------+------------------------------
Pure CSS animation        |   YES      |   YES
SVG animation (SMIL/CSS)  |   YES      |   YES
DOM + JS (rAF timers)     |   YES      |   YES
<canvas> 2D animation     |   YES      |   Often NO (drawImage blank
                          |            |   without preserve shim path)
WebGL / three.js          |   YES *    |   Usually NO
Video element playback    |   YES      |   NO (real video clock)
Web Audio-driven visuals  |   Partial  |   NO

* WebGL works in Quick mode now thanks to the preserveDrawingBuffer shim,
  provided the page is same-origin (srcdoc is, so this is the normal case).

7. ONE-LINE SUMMARY (for pasting into another AI)
--------------------------------------------------------------------------------

"In-browser tool that takes a user-uploaded HTML animation (CSS/SVG/DOM/
canvas/WebGL) and exports it as a real MP4 using WebCodecs + mp4-muxer,
with two modes: Quick (real-time snapshots, VFR timestamps, WebGL
preserveDrawingBuffer shim, html2canvas fallback for DOM) and High Quality
(virtual-time shim + frame-stepped deterministic render). Quick mode now
handles WebGL; High Quality is reliable for CSS/SVG/DOM but not for
canvas/WebGL/audio-driven animations. Remaining work: guardrails for
1080p60 long renders, memory cleanup, UI copy polish."
